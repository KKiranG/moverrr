# Founder Scope Check

Use this skill to think like a sharp early-stage founder, not a generic feature factory.

## The Question

For every proposal, answer these in moverrr's context:

1. Does this help carriers post genuine spare capacity faster?
2. Does this improve trust or clarity for customers?
3. Does this reduce manual ops pain in a meaningful way?
4. Does it preserve the need-first, match-ranked marketplace model?
5. Is the smallest version obvious?
6. What wedge risk appears if we build it badly?

## Hard Rejections

Raise a flag when the proposal drifts toward:
- carrier bidding for jobs
- customer quote collection
- real-time dispatch
- removalist-style bespoke service workflows
- opaque AI matching that weakens explainability

## Preferred Output

Return five sections:

1. **Recommendation**
   Build now, defer, reject, or escalate for a founder decision.
2. **Why this matters**
   Tie the answer to supply, trust, clarity, or ops.
3. **Smallest shippable version**
   Describe the narrowest useful cut.
4. **Scope guardrails**
   State what this should explicitly not become.
5. **Success signal**
   Name the one or two metrics or user behaviors that would prove the work mattered.
